## Project context (one paragraph)

Composer is a curated NYC date/night-out itinerary generator. Users answer a 5-step questionnaire (occasion → neighborhoods → budget → vibe → when) and get back a 2–4 stop **Opener → Main → Closer** evening. The product sits on a hybrid curation model: a human-curated venue database (founders Reid + Adit), a deterministic weighted scoring algorithm, and Gemini 2.5 Flash for copy polish only — never for venue selection. Built on Next.js 16 + Supabase, deployed at composer.onpalate.com via Vercel. Launch target: Columbia Business School community.

## Coding preferences (calibrated to Adit's style)

These are session-derived, not just from CLAUDE.md.

### Architecture

- **Single source of truth.** Every constant, threshold, taxonomy lives in one canonical module. Duplications are silent drift surfaces — fix them on sight.
- **Tunable knobs in `config.ts`.** `src/config/algorithm.ts` for scoring weights, `src/lib/venues/config.ts` for importer thresholds. No magic numbers in consuming files.
- **Layer separation.** `src/lib/` is logic. `src/app/` is UI + routes. UI imports types from `src/lib/`, never reaches into internals. Service-role Supabase client is server-only.
- **Discriminated unions over boolean flags** when state has more than two real variants. Established pattern: `{phase: 'loading'} | {phase: 'ready', data}`.
- **Schema migrations land with their writers.** If you add a NOT NULL column in a migration, you also update every INSERT call site in the same change. (Recurring bug pattern called out in saved memory.)

### Code style

- **TypeScript strict.** No `any`. No `ts-ignore`. Inline types for local-only shapes; `src/types/` for shared.
- **No comments unless WHY is non-obvious.** Document the surprising bit, the historical context, the workaround. Don't narrate WHAT (well-named identifiers do that).
- **Don't add error handling, fallbacks, or validation for impossible scenarios.** Trust internal code. Validate at boundaries (user input, external APIs).
- **Don't write defensive abstractions for hypothetical reuse.** If three call sites do something similar, that's not yet an abstraction. If it's the same thing in four places, it is.
- **Prefer editing existing files over creating new ones.**
- **No backwards-compat shims** unless explicitly asked. Just change the code.

### Brand voice (from BRAND_VOICE.md)

User-facing surfaces are observational, slightly dry, quietly funny. NOT chirpy, NOT stuffy, NOT performative. "For random Tuesdays" not "Level up your night!"

**Admin / failure / error states are an exception** — clinical-but-clear, no jokes. Operators making destructive decisions need precision over personality. `syncCopy.ts` is a model.

### Files / naming

- Components: `PascalCase.tsx`
- Utilities: `camelCase.ts`
- File name matches main export name
- Named exports everywhere except Next.js page files
- Imports: React → Next → third-party → internal (`@/lib`, `@/components`, `@/config`, `@/types`)

### Commits

- **One-line commit messages.** No multi-line bodies. No bullet lists. No co-author trailers. Format: `type(scope): description`.
- **Adit runs all git commands manually.** Provide the commit message and stop. Never run `git commit`, `git push`, `git add`. (Saved hard-rule memory.)

## Known gotchas / non-obvious things

These would catch a fresh agent off-guard.

1. **`docs/archive/` is historical, not authoritative.** When auditing, skip it. The four root docs (CLAUDE.md, ALGORITHM.md, README.md, BRAND_VOICE.md) are the truth.

2. **The `composer_apply_venue_import` Postgres function is `SECURITY DEFINER` with text-parameter dynamic SQL.** Safe ONLY because the TS caller in `apply.ts:buildSqlFragments` builds params from typed constants. Don't change `apply.ts` to accept user input or sheet data into those params without an explicit threat-model review. Migration file 20260502 has the full security note.

3. **`runApply` (high-level) goes through `applyPrepared`. CLI bypasses `runApply` and calls `prepareApply` + `applyPrepared` directly** so the interactive `[y/N]` prompt can interleave. Both paths use the same `safeRecord` audit recording. The override-empty-read guard lives in `applyPrepared` so both CLI and route get it.

4. **Audit recording is non-fatal.** `safeRecord` wraps `recordImportRun` in try/catch and logs warnings if the audit insert fails. Apply outcome is preserved; audit row may be lost. The route response includes `run_id: string | null` to surface this.

5. **`composer_import_runs` rows from Phase 2/3/4 (versions 20260501-20260503) have empty `statements` arrays** because they were applied via direct psql before the CLI workflow was unblocked. `supabase db push` accepts them. Only matters if anyone ever runs `db reset` (which is forbidden in this workflow). Followups #5 captures the optional fix.

6. **`composer_users` has RLS UPDATE blocked.** Profile edits go through `PATCH /api/profile` which validates server-side and writes via service-role. Never call `getBrowserSupabase().from('composer_users').update(...)` from the client — it'll silently fail RLS.

7. **`image_keys` is owned by enrichment**, not the importer. `PROTECTED_COLUMNS` in `src/lib/venues/columns.ts` formalizes this — the importer never writes it.

8. **HomeScreen wrapper has `flex flex-col` (load-bearing for `flex-1` on saved-plans)**. ProfilePage wrapper now matches. Don't remove `flex flex-col` from either without checking the descendants.

9. **Test migration `20260504000001_verify_migration_tracking_repair.sql` is intentionally on disk.** It's a `SELECT 1 WHERE FALSE` no-op, kept as durable evidence that `db push` works post-repair. Future ops engineers should NOT clean it up.

10. **`<img>` with eslint-disable comments** is a pre-existing pattern in `VenueLookup.tsx`, `StopCard.tsx`, `VenueDetailModal.tsx`. Lint flags them. Don't fix them ad-hoc — needs a coordinated `next/image` migration.

11. **The dev server was started by Adit at PID 27000.** It picks up file changes via Turbopack HMR. If you need to start one yourself it'll conflict on port 3000.
