## Proposed repair plan

The plan is two steps per orphan-duplicate file: (1) rename to a unique version, (2) insert a matching `schema_migrations` row marking it as already applied. The recorded-name file in each duplicate group stays untouched.

### Rename mapping

Strategy: bump each orphan to `YYYYMMDD000001`, `…000002` etc. — unique versions that sort immediately after the original date (sort order is irrelevant for already-applied migrations but reads naturally). The recorded-name file in each group keeps its existing version so the existing schema_migrations row keeps matching.

### `schema_migrations` actions

For each renamed file: insert a row marking the new version as already applied. Two equivalent options:

#### Option A (recommended) — direct INSERT via psql

Mirrors the proven Phase 2/3/4 pattern. Single SQL statement, no CLI dependency:

```sql
INSERT INTO supabase_migrations.schema_migrations (version, name, statements)
VALUES
  ('20260414000001', 'venue_data_import',  ARRAY[]::text[]),
  ('20260428000001', 'context_array',      ARRAY[]::text[]),
  ('20260428000002', 'venue_image_keys',   ARRAY[]::text[])
ON CONFLICT (version) DO NOTHING;
```

#### Option B — `supabase migration repair`

CLI-supported, calls roughly the same INSERT under the hood:

```bash
supabase migration repair --status applied 20260414000001
supabase migration repair --status applied 20260428000001
supabase migration repair --status applied 20260428000002
```

Either approach yields a row with `statements = ARRAY[]::text[]` — same shape as the existing Phase 2/3/4 rows that `db push --dry-run` already accepts.

| New version | Action | Command | Rationale |
|---|---|---|---|
| 20260414000001 | Mark as applied | INSERT (Option A) or `supabase migration repair --status applied 20260414000001` (Option B) | DDL (495 venue rows in `composer_venues`) is already in prod; record the migration as applied so `db push` doesn't try to re-run it. |
| 20260428000001 | Mark as applied | (same) | DDL (`composer_users.context` is `text[]`) is already in prod. |
| 20260428000002 | Mark as applied | (same) | DDL (`composer_venues_v2.image_keys` exists) is already in prod. |

**Recommended order of operations** (per file):
1. Rename file via `git mv` (preserves history).
2. INSERT corresponding row into `schema_migrations`.
3. Move on to next file.

If the file is renamed but the row is not yet inserted, `db push --dry-run` will flag the new version as "missing from history" (same status as today). That's a transient inconsistency, not a regression — the prod DB is no worse off than it currently is.

### Risk flags

- **🟢 LOW: All 3 orphan-duplicate DDLs are verifiably in production.** Checked column types, table existence, and row counts. The repair is purely cosmetic from the DB's perspective — no SQL is being applied or skipped.
- **🟢 LOW: New versions don't conflict.** Confirmed no rows exist for proposed versions. Confirmed no other files share these proposed names.
- **🟡 MEDIUM: Phase 2/3/4 workaround rows have `statements = ARRAY[]::text[]`.** This means the audit table doesn't have a faithful record of what those migrations did. `supabase db push` accepts them and doesn't complain. Out of scope for this phase but worth a note: the same shape is what `supabase migration repair` writes, so this is the canonical way to mark a row as "applied without re-running" in this CLI version. **No action needed.**
- **🟡 MEDIUM: The 20260304 `init` row has NULL statements.** Probably a Supabase project-init artifact (the file is 0 bytes). Pre-existed all this work. Don't touch — risk of unintended consequences if Supabase relies on this row's shape internally.
- **🟢 LOW: `git mv` for case-only or substring renames is a known macOS gotcha.** Not relevant here — these renames are length-changes, not case/substring tweaks. Standard `git mv` will work.
- **🟢 LOW: Order of file rename vs schema_migrations insert is forgiving.** Either order works; the worst case is a transient "missing from history" warning if the rename happens first, which `db push --dry-run` already shows today for these same files.
- **🟢 NONE: No content drift on already-recorded migrations.** Statement-count vs file-size comparison shows close match across all 20 cleanly-recorded rows. Nothing to reconcile.

### Verification (for eventual execute phase)

After renames + inserts:

1. `supabase db push --dry-run` should succeed without flagging any files. Expected output: "Remote database is up to date." or equivalent.
2. Create a no-op test migration:
   ```bash

   # body: SELECT 1 WHERE FALSE;  (or equivalent harmless statement)
   ```
3. `supabase db push` (without `--dry-run`). Should apply the test migration cleanly with no manual workaround.
4. Verify a row was inserted:
   ```sql

   ```
5. Cleanup: delete the test file. Optionally `DELETE FROM supabase_migrations.schema_migrations WHERE name = 'test_migration_tracking_repair';` to keep the table tidy. (`SELECT 1 WHERE FALSE;` is harmless to leave in the history table either way.)

If step 1 fails for any reason that isn't covered by the renames + inserts, **STOP** before proceeding to step 2. Investigate first.
