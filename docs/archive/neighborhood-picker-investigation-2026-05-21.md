# Neighborhood-picker filtering ‚Äî investigation

**Date:** 2026-05-21
**Trigger:** Need to know whether low-venue / no-venue neighborhoods are already being hidden from the questionnaire picker, and on what basis.

## 1. Files in play

- **Picker UI:** `src/components/shared/NeighborhoodPicker.tsx` ‚Äî the shared pill-selection component. Two consumers: `NeighborhoodStep` (questionnaire) and the now-commented-out onboarding step.
- **Questionnaire wrapper:** `src/components/questionnaire/NeighborhoodStep.tsx` ‚Äî adds the "Pick up to 3" helper, prefill, and Continue button, but **does no additional filtering** of its own.
- **Data source:** `src/config/neighborhoods.ts:75` re-exports the 25 groups from `src/config/generated/neighborhoods.ts` ‚Äî auto-generated by `npm run generate-configs`.

## 2. Answer to (a/b/c/d): **(b) Filtered at picker render time by a venue-count check.**

`NeighborhoodPicker.tsx:66-69`:

```ts
const minVenues = ALGORITHM.pools.minGroupVenuesToRender;
const filtered = NEIGHBORHOOD_GROUPS.filter((g) => g.venueCount >= minVenues);
// Fallback to unfiltered if threshold hides everything
const groups = filtered.length > 0 ? filtered : NEIGHBORHOOD_GROUPS;
```

So filtering exists and runs every render before any UI is built. Fallback: if every group falls below the threshold (defensive), the picker shows ALL groups unfiltered ‚Äî a guard against a misconfigured threshold, not an active code path in current calibration.

## 3. Threshold + semantics

**Threshold value:** `src/config/algorithm.ts:241`:
```ts
minGroupVenuesToRender: 50,
```
With a JSDoc comment at `algorithm.ts:229-240` calling out "Calibrated 2026-04-29 at 50" and a sane range of 30‚Äì75.

**What it counts:** `g.venueCount` ‚Äî the field is **total active venues per group**, baked into the generated config at sheet-sync time. Source pipeline:
- `scripts/generate-configs.py:96-120` ‚Äî `fetch_venue_counts_by_neighborhood()` queries `composer_venues_v2` with the single filter `.eq("active", True)`. No occasion-tag filter, no stop-role filter, no business-status filter.
- Counts are summed across each group's storage slugs at `generate-configs.py:462` and emitted into `src/config/generated/neighborhoods.ts` as the `venueCount` field on each group.

**Bucket-awareness:** **None.** The threshold applies the same total-active-venue count regardless of whether the user has selected Date / Friends / Solo.

**When it applies:** **Before the user picks a bucket.** Looking at `QuestionnaireShell.tsx`, the step order is `occasion ‚Üí neighborhoods ‚Üí budget ‚Üí vibe ‚Üí when`. So the picker renders *after* the user has picked occasion in flow order, but the filter itself never reads `state.answers.occasion` ‚Äî it only reads `g.venueCount` and the constant threshold. Bucket choice is irrelevant to which neighborhoods appear.

**Staleness note:** `venueCount` is a snapshot, regenerated only when someone runs `npm run generate-configs`. The current `src/config/generated/neighborhoods.ts` was generated 2026-05-01. Live counts have drifted ‚Äî for example, **Koreatown has `venueCount: 3` baked in but the live DB has 41 active venues there** (verified 2026-05-20 via SQL). The picker uses the stale `3`, so Koreatown is hidden whether or not 41 ‚â• 50.

## 4. Harlem walkthrough

From `generated/neighborhoods.ts`: `harlem_uptown.venueCount = 5`. Threshold = 50. **5 < 50 ‚Üí HIDDEN.**

Across all three buckets:
- **Date:** Hidden. Filter is bucket-agnostic; doesn't look at the 5 Date-tagged venues mentioned in the preflight.
- **Friends:** Hidden. Same reason.
- **Solo:** Hidden. Same reason.

So the 5-Date-venue concern flagged in the preflight is moot for the live picker ‚Äî Harlem never appears regardless of bucket. The preflight's risk flag was theoretical; in practice the venueCount threshold already cuts it off.

## 5. Koreatown + Queens ‚Äî does the "zero closers" issue slip through?

**No, but only by coincidence.** Both are hidden because their `venueCount` is below 50, not because the filter understands stop-role coverage.

| Group | Baked `venueCount` | Live total (active) | Live closer-eligible Date | Picker status |
|---|---:|---:|---:|---|
| Koreatown | 3 | 41 | **0** | Hidden (count<50) |
| Queens | 25 | 25 | **0** | Hidden (count<50) |

The filter at `NeighborhoodPicker.tsx:67` only checks `g.venueCount >= minVenues`. **It does not inspect occasion tags, stop_roles, or any role expansion.** A hypothetical group with `venueCount ‚â• 50` and zero closer-eligible venues would slip through and surface in the picker, then fall back on cascade relaxation at composition time when no closer can be found.

In current data, none of the 7 picker-visible groups (West Village, East Village/LES, SoHo/Nolita/Tribeca, Flatiron/NoMad, Williamsburg/Greenpoint, East Williamsburg/Bushwick, DUMBO/Brooklyn Heights) have the closer-zero issue, so the gap doesn't bite today. But the gap is real and would bite if a future high-volume group lacked closer-eligible venues.

## 6. Threshold-value search

`grep -rEn 'minGroupVenuesToRender|venueCount|min_venues|minimumVenues|threshold' src/`:

- `ALGORITHM.pools.minGroupVenuesToRender` ‚Üí only defined at `algorithm.ts:241`, read at `NeighborhoodPicker.tsx:66`. One definition, one consumer.
- `venueCount` ‚Üí field on the `NeighborhoodGroup` interface at `generated/neighborhoods.ts:8`; plumbed through `src/config/neighborhoods.ts:76`; read only at `NeighborhoodPicker.tsx:67`.
- **No `25` literal** anywhere related to neighborhood filtering. (The bucket-distribution recon flagged a "<8 venues" risk threshold, which was an analytical heuristic, not a code constant.)
- The other `threshold` hits in the codebase (`applyThresholdBlocked`, `ThresholdOverrideDialog`, etc.) are venue-import threshold guards ‚Äî unrelated to picker filtering.

## Net read

The picker filters by total active venue count, threshold 50, bucket-agnostic and role-agnostic. Threshold lives in `algorithm.ts`. It hides 18 of the 25 groups today (effectively keeps West Village, EV/LES, SoHo cluster, Flatiron/NoMad, Williamsburg, EWburg/Bushwick, DUMBO/Brooklyn Heights). The preflight's "<8 Date venues" and "0 closers" concerns for Harlem/Koreatown/Queens are not actively risks at the picker level ‚Äî those groups never appear ‚Äî but the underlying logic has two real gaps worth knowing about:

1. **Staleness:** `venueCount` only updates on `npm run generate-configs`. Live drift (e.g. Koreatown 3 ‚Üí 41) is invisible to the filter.
2. **No role coverage check:** A group with ‚â•50 total venues but zero closer-eligible venues would still appear in the picker; the closer-zero failure would only manifest at composition time via cascade relaxation.
