What this requires on the backend
  1. New pinnedStops array on the plan request: [{ slot: 1, placeId, name, lat, lng }, { slot: 3, placeId, name, lat, lng }]
  2. planStopMix sees the pin count and locks those slots in place when building the category plan. Unpinned slots fill the remaining categories.
  3. The Google Places cascade skips pinned slots entirely and only runs for unpinned ones. The anchor for an unpinned slot becomes the nearest adjacent pinned or already-resolved stop.
  4. Walking-cap check runs against both neighbors. If the user pins two places that are more than the cap apart, show a warning in the plan setup (not a hard block — user choice wins).
  5. New /api/places/autocomplete endpoint wrapping Google Places Autocomplete, biased to the selected neighborhoods.
Estimated effort: ~1 day of implementation. The trickiest part isn't the UI — it's making the anchor graph right when stops can resolve out of order (e.g., pin 1 + pin 3, fill 2 between them).
